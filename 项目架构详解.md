# 项目架构详解

> Hermes to-B Agent 的完整架构说明：分层、模块职责、数据流、存储、关键决策。
> 配合 [`改造计划.md`](改造计划.md)（9 个架构决策）+ [`API文档.md`](API文档.md)（接口）阅读。

## 1. 总体架构

这是一个**单租户、每客户独立部署**的 to-B 智能体框架。一个客户 = 一套部署（on-prem 或专属 VPC），客户内的 10–50 个用户**共享一个实例**，但会话/记忆/权限按 `user_id` 严格隔离。

```
┌─────────────┐   HTTP/SSE    ┌──────────────────────────────────┐
│  你的前端   │ ────────────> │  server/  (headless FastAPI BFF) │
│ (Web/小程序)│ <──────────── │  JWT 鉴权 · 多用户隔离           │
└─────────────┘   SSE 流      └───────────────┬──────────────────┘
                                               │ 每会话构造一个
                                               ▼
                              ┌────────────────────────────────┐
                              │  AIAgent (run_agent.py)        │
                              │  核心对话循环 + 工具调用        │
                              └───┬────────┬──────────┬────────┘
                                  │        │          │
                          db_query│  terminal│     plan/mem
                                  ▼        ▼          ▼
                          ┌──────────┐ ┌─────────┐ ┌────────┐
                          │客户业务库│ │Docker   │ │注入提示│
                          │ (只读)   │ │沙盒     │ │词/会话 │
                          └──────────┘ └─────────┘ └────────┘

数据层（全在 $HERMES_HOME）:  state.db(会话) · memory.db(记忆) · users.db(用户) · jwt.key
```

### 分层

| 层 | 职责 | 代码位置 |
|---|---|---|
| **接入层** | HTTP/SSE、JWT 鉴权、多用户隔离、请求路由 | `server/` |
| **Agent 核心层** | 对话循环、工具调用编排、提示词构建、上下文压缩 | `run_agent.py` + `agent/` + `model_tools.py` |
| **能力层（工具）** | db_query、terminal/execute_code、todo、delegate 等 | `tools/` |
| **执行后端层** | Docker 沙盒（代码执行）、本地工具函数 | `tools/environments/` |
| **持久层** | 会话(SessionDB)、记忆、用户——全 SQLite | `hermes_state.py` + `server/{auth,memory,sessions}.py` |
| **配置/基础设施** | 路径解析、配置加载、模型 provider | `hermes_constants.py` + `hermes_cli/config.py` + `plugins/model-providers/` |

## 2. 数据流：一个 `/chat` 请求的完整生命周期

以「用户问：统计客户表里有多少行」为例：

1. **前端** `POST /chat`，Header 带 `Authorization: Bearer <JWT>`，Body `{message:"统计客户表行数", session_id?: "...", mode?: "plan"|"execute"}`。
2. **`server/routes/chat.py`** → FastAPI 依赖 `get_current_user` 解 JWT → 得到 `user_id`。
3. 若带 `session_id` → `server/sessions.py::assert_session_owned` 校验该会话属于此用户（防越权，403 否则）。
4. 起一个**工作线程**跑 `run_agent()`：
   - `server/sessions.py` 确保会话行存在（`SessionDB.create_session`，source=`headless`，user_id）。
   - `get_messages_as_conversation` 加载历史 → 作为 `prefill_messages`（跨会话续上下文）。
   - `server/agent_factory.build_agent` 构造 `AIAgent`（provider/model/reasoning_config/toolsets/session_id/user_id/ephemeral_system_prompt）。
   - `agent.chat(message, stream_callback=on_delta)` —— 同步跑核心循环：
     - AIAgent 发请求给 DeepSeek（经 provider profile），模型决定调 `db_query`。
     - `model_tools.handle_function_call` 执行 `db_query(sql)` → 连客户库（只读凭证）→ 返回行数 JSON。
     - 模型拿到结果，生成自然语言回答「共 5 行」。
     - 每个 token 经 `stream_callback` → 推入 queue。
   - 完成后 `SessionDB.append_message` 存 user + assistant 消息（下次可续）。
5. **SSE 流**回前端：`delta`（逐 token）→ `final`（完整回答）→ `done`（带 session_id + user_id）。

整个过程中，agent 若调 `terminal`/`execute_code` → 路由到 **Docker 沙盒**（`tools/environments/docker.py` 起容器跑命令），**够不到宿主机**。

## 3. 模块详解

### 3.1 接入层 `server/`（headless FastAPI 服务）

这是**改造的核心新增**——你前端的 BFF。所有用户请求经此进入。

| 文件 | 职责 |
|---|---|
| `server/__main__.py` | 入口（`python -m server`）：校验 `HERMES_HOME` → 加载 `.env` → `uvicorn.run`（host/port 走 env，容器内绑 0.0.0.0） |
| `server/app.py` | FastAPI 应用工厂 `create_app()`：启动时 `auth.init_db`（bootstrap admin）+ `memory.init_db` + `features.apply_terminal_backend`（按开关定 TERMINAL_ENV）+ 挂载所有路由 + `/health` |
| `server/auth.py` | 鉴权：`users.db`（SQLite + bcrypt 哈希）+ JWT（PyJWT，HS256，24h）+ 首跑 bootstrap admin + 用户的 CRUD |
| `server/deps.py` | FastAPI 依赖：`get_current_user`（Bearer→JWT→user）、`require_admin`（role 校验） |
| `server/features.py` | 功能开关：`get_features()`（读 config.yaml `features` 段 + `HERMES_FEATURE_*` env）+ `apply_terminal_backend()`（host_terminal 决定 docker/local） |
| `server/memory.py` | 持久记忆：`memory.db`（SQLite，按 user_id），save/list/delete |
| `server/sessions.py` | 会话管理：封装 SessionDB，按 user_id 列/取会话、验归属（防越权） |
| `server/agent_factory.py` | **AIAgent 构造器**：组装 provider/model/reasoning/toolsets/session_id/user_id/prefill(历史)/ephemeral_system_prompt(记忆+plan) → 一个 `AIAgent` |
| `server/routes/auth.py` | `POST /auth/login`、`POST /auth/register`(admin)、`GET /auth/me` |
| `server/routes/chat.py` | `POST /chat`（SSE 流式，工作线程 + queue） |
| `server/routes/sessions.py` | `GET /sessions`、`GET /sessions/{id}`、`POST /sessions/{id}/resume` |
| `server/routes/memory.py` | `GET/POST/DELETE /memory`（按用户） |
| `server/routes/users.py` | `GET/POST/DELETE /users`、`PUT /users/{id}/role`（全 admin） |
| `server/routes/features.py` | `GET /features`（前端读开关状态） |

### 3.2 Agent 核心层

| 文件 | 职责 |
|---|---|
| `run_agent.py` | **`AIAgent` 类**——核心对话循环（`run_conversation`：模型调用→工具调用→结果回灌→循环，受 `max_iterations`/预算约束）。`chat()` 是简单接口（`stream_callback` 推 token）。~60 个构造参数（provider/model/工具集/会话/预算/回调…） |
| `model_tools.py` | 工具编排：`discover_builtin_tools()`（导入 `tools/*.py` 触发注册）、`handle_function_call()`（执行工具 + 返回 JSON）。插件 hook 也在此 |
| `toolsets.py` | `TOOLSETS` 字典（工具集定义）+ `_HERMES_CORE_TOOLS`（默认捆绑）。工具必须既注册**又**在某个 toolset 里才会暴露给 agent |
| `agent/agent_init.py` | `AIAgent.__init__` 的实际实现（构造器转发到此）——设置 session_db、reasoning_config、回调、provider 解析等 |
| `agent/conversation_loop.py` | 对话循环的细节（工具调用、中断检查、预算） |
| `agent/context_compressor.py` | 上下文压缩（对话太长时压缩历史，**唯一允许破坏 prompt cache 的场景**） |
| `agent/session_context.py` | （从 gateway 抽出）ContextVar 会话环境桥接——把 terminal.cwd 等配置桥接到子进程 env |
| `agent/curator.py` | **自进化 skill 引擎**（评审循环 + 自动归档老化 skill）——保留，待 3.4 改造 |

### 3.3 能力层（工具）`tools/`

工具通过 `tools/registry.py::registry.register()` 注册，`model_tools` 自动发现。

| 工具文件 | 工具 | 作用 |
|---|---|---|
| `tools/db_query.py` | `db_query` | **查客户业务库**（HERMES_DB_URL），只读在 DB GRANT 层强制。返回 JSON（columns/rows） |
| `tools/terminal_tool.py` | `terminal` / `process` | 跑 shell 命令——路由到 `_create_environment`（local/docker 二选一） |
| `tools/code_execution_tool.py` | `execute_code` | **PTC**（Programmatic Tool Calling）：agent 写 Python 脚本，脚本里 RPC 调工具，在沙盒里跑。把多步工具链压成一次推理 |
| `tools/delegate_tool.py` | `delegate_task` | **sub-agent**：派生子 AIAgent（隔离上下文 + 终端），支持单/批量并行 |
| `tools/todo_tool.py` | `todo` | 内置任务规划跟踪 |
| `tools/mcp_tool.py` | （MCP 客户端） | 连接外部 MCP server，拿更多工具（决策 6：你的算法/数据接口做成 MCP） |
| `tools/skills_*.py` | skill 管理 | skill 加载/同步/守卫/审计/溯源——为 3.4 本地 skill-hub 保留 |

### 3.4 执行后端层 `tools/environments/`

`terminal`/`execute_code` 在此跑代码。**决策 5：只留 local（工具枢纽）+ docker（沙盒）。**

| 文件 | 角色 |
|---|---|
| `tools/environments/docker.py` | **Docker 沙盒后端**：起容器（TERMINAL_DOCKER_IMAGE）跑命令，持久文件系统、资源限制（cpus/memory/pids，**storage-opt 已禁用**因 Docker Desktop 不支持） |
| `tools/environments/local.py` | **本地后端 + 工具枢纽**：LocalEnvironment 类 + 一堆通用子进程工具函数（`_sanitize_subprocess_env`/`hermes_subprocess_env`/`_find_shell` 等，被 cron/cli/agent 多处引用）。**默认不用作终端后端**（TERMINAL_ENV=docker），除非 `features.host_terminal` 开启 |
| `tools/environments/base.py` | BaseEnvironment ABC |
| `tools/environments/file_sync.py` | 文件同步到远程后端 |

> ssh/modal/managed_modal/daytona/singularity 已在 Step 2.4 删除——agent 物理上只能进 docker 沙盒（或开启 host_terminal 时的 local）。

### 3.5 持久层

| 文件 | 数据 | 说明 |
|---|---|---|
| `hermes_state.py` (`SessionDB`) | `state.db` | 会话 + 消息（SQLite + **FTS5 全文检索**）。`create_session`/`get_messages_as_conversation`/`append_message`/`list_sessions_rich`。按 (source, user_id, chat_id) 索引 |
| `server/auth.py` | `users.db` | 用户（id/username/bcrypt 哈希/role） |
| `server/memory.py` | `memory.db` | 持久记忆（按 user_id） |
| `agent/memory_provider.py` + `memory_manager.py` | — | MemoryProvider **ABC + 编排器**（保留，无 provider 安装；3.4 可实现本地 provider 接入） |

**所有数据都在 `$HERMES_HOME`**（dev=`.hermes-dev`，容器=`/data`）。本地 SQLite，单租户部署下存在客户环境内。客户的**业务数据**则在其自己的数据库（HERMES_DB_URL 连），不在 HERMES_HOME。

### 3.6 配置/基础设施

| 文件 | 职责 |
|---|---|
| `hermes_constants.py` | `get_hermes_home()`/`display_hermes_home()`（**所有路径必须走它，不能硬编码 ~/.hermes**）+ `parse_reasoning_effort()`（none→{"enabled":False} 关 thinking） |
| `hermes_cli/config.py` | `DEFAULT_CONFIG` + `load_config()`（config.yaml 加载）+ `OPTIONAL_ENV_VARS`（.env 密钥清单） |
| `hermes_logging.py` | 日志（agent.log/errors.log，profile 感知，Windows 用 concurrent-log-handler） |
| `plugins/model-providers/` | **4 个保留 provider**：`custom`（自托管 OpenAI 兼容）、`zai`（智谱 GLM）、`alibaba`（百炼 Qwen）、`deepseek`。每个一个 ProviderProfile（env 变量/base_url/模型）。惰性发现 |
| `hermes_cli/main.py` | `hermes` 命令入口（admin 子命令：setup/tools/cron/config/mcp/curator/model/memory 等；已中和 acp/gateway/profile/gui/dashboard 等已删表面） |
| `cron/scheduler.py` | 定时任务调度（保留；messaging 投递已 stub，投递走 headless 服务） |

## 4. 关键设计决策（摘要）

完整 9 个决策见 [`改造计划.md`](改造计划.md)。最影响架构的：

1. **单租户每客户独立部署** + 共享实例多用户 → 代码无租户概念，靠 user_id 隔离。
2. **沙盒默认**（docker-only）→ agent 够不到宿主；`features.host_terminal` 是唯一 opt-in。
3. **DB 只读在 GRANT 层** → 工具不限制 SQL，数据库用只读用户拒绝写。
4. **能力面分层**：db_query 核心工具；算法/数据接口按规则做核心工具或 MCP server；专门 agent（如 nl2sql）做 MCP 包裹服务；临时并行用 delegate_task。
5. **持久记忆本地化**（无云 provider）+ 按 user_id 隔离。
6. **prompt cache 神圣**（继承自 Hermes）→ 不在对话中途改上下文/换工具集/重建系统提示。
7. **硬分叉不跟踪上游** → 安全修复手动 cherry-pick（见 UPSTREAM.md）。

## 5. 改造中已删除/软禁用的（避免混淆）

- **已物理删除**：TUI(ui-tui/tui_gateway)、desktop(apps)、dashboard(web/web_server)、ACP(acp_adapter)、消息网关平台适配器(gateway 的 20 个 platform，gateway 整体软禁用)、~24 个模型 provider、8 个云 memory provider、5 个终端后端(ssh/modal/managed_modal/daytona/singularity)、研究工具(batch_runner/trajectory_compressor/mcp_serve 等)、OSS 脚手架(nix/.github/CONTRIBUTING 等)、bundled skills 数据、旧 Docker。
- **软禁用（代码留，功能关）**：computer_use + kanban（check_fn 门控，默认不激活）；gateway 命令（handler 中和报「已移除」）。
- **`local.py` 保留**：是通用子进程工具枢纽（被多处引用），不只是终端后端。
