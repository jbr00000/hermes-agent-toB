## 1. 总体架构

这是一个**单租户、每客户独立部署**的 to-B 智能体框架。一个客户 = 一套部署（on-prem 或专属 VPC），客户内的 10–50 个用户**共享一个实例**，但会话/记忆/权限按 `user_id` 严格隔离。

```
┌─────────────┐   HTTP/SSE    ┌──────────────────────────────────┐
│  你的前端   │ ────────────> │  server/  (headless FastAPI BFF) │
│ (Web/小程序)│ <──────────── │  JWT 鉴权 · 多用户隔离           │
└─────────────┘   SSE 流      └───────────────┬──────────────────┘
                                               │ 每会话构造一个
                                               ▼
                              ┌────────────────────────────────┐
                              │  AIAgent (run_agent.py)        │
                              │  核心对话循环 + 工具调用        │
                              └───┬────────┬──────────┬────────┘
                                  │        │          │
                          db_query│  terminal│     plan/mem
                                  ▼        ▼          ▼
                          ┌──────────┐ ┌─────────┐ ┌────────┐
                          │客户业务库│ │Docker   │ │注入提示│
                          │ (只读)   │ │沙盒     │ │词/会话 │
                          └──────────┘ └─────────┘ └────────┘

数据层（全在 $HERMES_HOME）:  state.db(会话) · memory.db(记忆) · users.db(用户) · audit.db(审计) · jwt.key
```

### 分层

| 层 | 职责 | 代码位置 |
|---|---|---|
| **接入层** | HTTP/SSE、JWT 鉴权、多用户隔离、请求路由 | `server/` |
| **Agent 核心层** | 对话循环、工具调用编排、提示词构建、上下文压缩 | `run_agent.py` + `agent/` + `model_tools.py` |
| **能力层（工具）** | db_query、terminal/execute_code、todo、delegate 等 | `tools/` |
| **执行后端层** | Docker 沙盒（代码执行）、本地工具函数 | `tools/environments/` |
| **持久层** | 会话(SessionDB)、记忆、用户——全 SQLite | `hermes_state.py` + `server/{auth,memory,sessions}.py` |
| **配置/基础设施** | 路径解析、配置加载、模型 provider | `hermes_constants.py` + `hermes_cli/config.py` + `plugins/model-providers/` |

## 2. 数据流：一个 `/chat` 请求的完整生命周期

以「用户问：统计客户表里有多少行」为例：

1. **前端** `POST /chat`，Header 带 `Authorization: Bearer <JWT>`，Body `{message:"统计客户表行数", session_id?: "...", mode?: "plan"|"execute"}`。
2. **`server/routes/chat.py`** → FastAPI 依赖 `get_current_user` 解 JWT → 得到 `user_id`。
3. 若带 `session_id` → `server/sessions.py::assert_session_owned` 校验该会话属于此用户（防越权，403 否则）。
4. 起一个**工作线程**跑 `run_agent()`：
   - `server/sessions.py` 确保会话行存在（`SessionDB.create_session`，source=`headless`，user_id）。
   - `get_messages_as_conversation` 加载历史 → 作为 `prefill_messages`（跨会话续上下文）。
   - `server/agent_factory.build_agent` 构造 `AIAgent`（provider/model/reasoning_config/toolsets/session_id/user_id/ephemeral_system_prompt）。
   - `agent.chat(message, stream_callback=on_delta)` —— 同步跑核心循环：
     - AIAgent 发请求给 DeepSeek（经 provider profile），模型决定调 `db_query`。
     - `model_tools.handle_function_call` 执行 `db_query(sql)` → 连客户库（只读凭证）→ 返回行数 JSON。
     - 模型拿到结果，生成自然语言回答「共 5 行」。
     - 每个 token 经 `stream_callback` → 推入 queue。
   - 完成后 `SessionDB.append_message` 存 user + assistant 消息（下次可续）。
5. **SSE 流**回前端：`delta`（逐 token）→ `final`（完整回答）→ `done`（带 session_id + user_id）。

整个过程中，agent 若调 `terminal`/`execute_code` → 路由到 **Docker 沙盒**（`tools/environments/docker.py` 起容器跑命令），**够不到宿主机**。

## 4. 核心运行机制：多轮对话循环 + 上下文压缩

> §2 走了一遍 `/chat` 一轮的高层数据流。这一节拆开讲两个最关键的**内部机制**：agent 怎么"一轮接一轮聊下去"，以及长对话怎么"不撑爆"。代码事实带 `file:line`；真正实现都在 `agent/` 子模块——`run_agent.py` 里的 `run_conversation` / `chat` / `__init__` 都只是转发器。

### 4.1 多轮对话循环

真正的循环在 `agent/conversation_loop.py:518`，不在 `run_agent.py`。

#### （1）核心循环长什么样

一句话：**问模型 → 模型要调工具就调、把结果塞回去再问 → 模型不调工具了，这次的回答就是最终回答**。

循环骨架（`agent/conversation_loop.py:633`）：

```python
while (迭代没超上限 and 预算还有) or 宽限一次:
    调模型，拿到 assistant 的回复
    if 回复里有 tool_calls:          # 模型说"我要调工具"
        把 assistant 这条消息存下来
        执行工具，把每个工具结果也存下来
        continue                     # ★ 回到循环顶，带着工具结果再问模型
    else:                            # 模型没要调工具，直接给了文字回答
        把这条最终回答存下来
        break                        # 退出循环
```

精髓是那个 `continue`：只要模型还想调工具，就把工具结果挂到上下文里、再问一遍；模型哪次不调工具了，那句话就是给用户的最终回答，循环结束。

> **一句话总结**：循环不是"调一次模型就完"，而是"模型自己决定要不要继续调工具"——它要，就满足它再问一轮；它不要，就收工。

#### （2）一轮对话里，消息列表怎么"长"出来

消息用标准 **OpenAI 聊天格式**，每条带个角色 `role`。一轮带 2 个工具调用的对话，消息列表是这样长大的：

```
开始：  [user] 统计客户表行数

模型说要调 db_query：
        [user]      统计客户表行数
        [assistant] (tool_calls: db_query)   ← 模型这条先存

执行工具，结果回来：
        [user]      统计客户表行数
        [assistant] (tool_calls: db_query)
        [tool]      {"rows": 5}              ← 工具结果存成 role=tool

带着结果再问模型，模型给出文字：
        [user]      统计客户表行数
        [assistant] (tool_calls: db_query)
        [tool]      {"rows": 5}
        [assistant] 共 5 行                  ← 最终回答，循环结束
```

下一轮 API 调用就把**整条列表**发回模型。所谓"多轮"，就是这条列表不断 append。

#### （3）为什么角色永远不会"撞车"

CLAUDE.md 有条铁律：**绝不连续两条相同角色的消息**（比如两条 `user` 或两条 `assistant` 连着）。保证这点的关键是一个细节——

> **工具结果的 `role` 是 `"tool"`，不是 `"user"`。**

OpenAI 协议里 `tool` 和 `user` 是两个独立角色，所以工具链 `assistant → tool → tool → assistant` 天然交替，绝不会出现 `user → user`。代码里发请求前还有一道 `repair_message_sequence` 兜底（`conversation_loop.py:773`），万一历史里混进了违规尾巴，自动修。

> **一句话总结**：工具结果独占一个 `tool` 角色，跟 `user` / `assistant` 都不撞，这是角色交替不破的根本原因。

#### （4）多个工具调用：并发还是排队

**默认并发**。模型一轮里要调多个工具时，只要这批工具互不冲突（都是只读、或写的是不同文件），就丢进线程池一起跑（`agent/tool_executor.py:306`，最多 8 个并发）。但有个贴心细节：

> **结果按模型给出的顺序填回，不按谁先跑完。**

这样模型看到的工具结果顺序，和它发出调用的顺序一致，不会因为并发而错位。只有互相冲突的工具（比如改同一个文件）才退化成排队。

#### （5）怎么防止 agent 陷入死循环

三层闸门：

| 闸门 | 意思 | 默认 |
|---|---|---|
| `max_iterations` | 单个 agent 最多调模型多少次 | 90 |
| `iteration_budget` | 父子 agent **共享**的预算（子 agent 不能绕过父亲无限跑） | 父传子 |
| `_budget_grace_call` | 预算耗尽时塞一条"总结一下"提示，**再给最后一次机会** | 用完即失效 |

另外 `execute_code`（在沙箱里跑一段 Python）这种廉价调用会**退还**预算，不占额度。

> **一句话总结**：硬上限 + 父子共享预算 + 一次宽限，三道关保证 agent 不会无限循环烧钱。

#### （6）模型的"思考过程"存哪

很多模型（DeepSeek-R1、Kimi thinking 等）会先输出一段推理再给答案。这段思考：

- **内部**存在 `assistant_msg["reasoning"]` 字段（给历史/显示/调试用）。
- **发请求时**拷贝成 `reasoning_content` 字段，然后**删掉** `reasoning` 字段本身（`conversation_loop.py:806`）——因为接口不认 `reasoning` 这个键。
- **落库**时 `reasoning` 是独立列，resume 能完整还原。

有个坑：DeepSeek-v4 thinking、Kimi、MiMo 这些模型，**回放带 tool_calls 的历史时必须带 `reasoning_content`**，否则 HTTP 400。代码用 `_needs_thinking_reasoning_pad()` 检测并自动补上。

#### （7）跨会话：新一轮怎么"接着上次的聊"

to-B 服务端的做法有点特殊（`server/agent_factory.py:20`）：**它不让 `AIAgent` 自己管历史，而是每个 `/chat` 请求来，都从 `state.db` 把历史现拉出来，塞给一个全新的 agent。**

具体流程（`server/routes/chat.py`）：

```
1. 收到带 session_id 的请求
2. db.get_messages_as_conversation(session_id)   ← 从 SQLite 把历史拉成 OpenAI 格式
3. build_agent(prefill_messages=历史, ...)        ← 把历史当"预填消息"塞给新 agent
4. agent.chat(本次消息)
5. db.append_message("user",     本次消息)       ← 落库
   db.append_message("assistant", 最终回答)      ← 落库
```

`prefill_messages` 怎么进上下文？每次调模型时，它被插在 **system 提示之后、本轮新消息之前**（`conversation_loop.py:871`），而且**它不进 agent 自己的消息列表、不落库**——纯粹是这次请求的临时拼接。

> ⚠️ **一个值得注意的细节**：服务端只把**最终的 user 文本 + assistant 文本**落库（`chat.py:118-119`），中间的工具调用、工具结果、reasoning **都不落库**。所以 resume 时历史里只有"历轮提问 + 历轮最终回答"，没有工具链。对连续性够用；但如果想 resume 后还能看到上轮查了什么 SQL，要改的就是落库那一行 + `get_messages_as_conversation` 的过滤。

#### （8）为什么 system prompt 有个"临时"版本

system prompt 分两层：

- **`_cached_system_prompt`**：一次会话构建一次，落库，后续**逐字节复用**——这是命中 prompt cache 的基础，绝不在会话中途改。
- **`ephemeral_system_prompt`（临时）**：每次 `/chat` 请求现拉的"用户记忆 + plan 模式指令"，临时拼到 cached 后面发出去，**不落库、不进消息列表**。

为什么要分两层？因为"在会话中途改 system prompt"是破坏 cache 的大忌。服务端绕开它的办法是：**每个请求都新建一个 agent**，对 agent 来说"每次都是会话第一轮"，靠"每次请求重读 SQLite 刷新记忆"来更新，而不是"在同一个会话里改 system prompt"。

> **一句话总结**：系统提示有"固定底座"（保 cache）和"临时贴纸"（每次请求刷新记忆）两层；靠每请求重建 agent 来规避"会话中途改提示"的禁忌。

### 4.2 上下文压缩

> 长对话 token 会超模型上限。压缩 = 把旧对话缩成摘要腾空间。但它必须**不破坏上面 4.1(3) 的角色交替铁律**，还要**尽量少破坏 prompt cache**——所以它被当成"最后手段"严格管控。真正逻辑在 `agent/context_compressor.py` + `agent/conversation_compression.py`。

#### （1）什么时候触发

**按 token 数，不是按消息条数**。规则（`context_compressor.py:871`）：

```
有效输入预算 = 模型窗口 - 输出预留(max_tokens)
触发阈值     = max(有效预算 × 50%, 64000)
```

两个保护：大模型不被拉低到 6.4 万（200K 窗口就在 10 万触发，而非 6.4 万）；小模型不会卡死（窗口正好等于阈值时改用 85% 触发，保证压在 provider 拒绝前先跑）。

还有两个"尽量别压"的闸：**摘要模型刚失败会冷却 10 分钟**；**连续两次压缩省不到 10% 就跳过**（提示用户开新会话）。手动 `/compress` 可绕过冷却。

触发时机有两处：每轮工具循环后、下次调模型前的**主动预检**；provider 已经因超限拒绝时的**错误恢复强制压缩**（最多重试 3 次）。

> **一句话总结**：token 到阈值就压；但冷却期、反抖动、provider 拒绝前主动预检三重机制，让它尽量少发生。

#### （2）压缩分几步走

入口 `compress()`（`context_compressor.py:2630`），五步：

1. **廉价预剪枝（不调 LLM）**：去重重复的工具结果；把又长又旧的工具结果（比如 terminal 输出）换成一行摘要 `ran npm test → exit 0`；截断过长的工具调用参数。这步免费，先省一波。
2. **定边界**：决定**头部留几条、尾部留几条**（尾部按 token 预算留，保证最新的用户提问和回答不被卷走），边界还要对齐 tool_call / result 配对，不能切断。
3. **LLM 结构化摘要**：把中间那段喂给摘要模型，产出固定结构（Historical Task Snapshot / Completed Actions / Active State / Blocked / Key Decisions / Relevant Files / Critical Context 等）。多次压缩时用"迭代更新"模式，把上次摘要当上下文做增量，不丢累积信息。
4. **拼回去**：头部原文 + 摘要 + 尾部原文（详见下一节）。
5. **收尾清理**：删孤儿 tool_call / result；旧截图占位化（不反复重发几 MB base64）。

所有内容喂摘要器前先过 `redact_sensitive_text` 去 secret；摘要用与用户一致的语言（用户说中文就中文摘要）。摘要前后还强制加固定前缀/后缀，防止弱模型把摘要里的引文当成新指令执行。

> **一句话总结**：先免费剪一波（预剪枝），再定切分边界，中间段调 LLM 产结构化摘要，最后拼回去——能省 LLM 就省。

#### （3）压缩后怎么替换上下文（不破坏角色交替）

**压缩不是删旧消息换一条，而是把消息列表切成三段：**

```
[head 头部]   开头几条原样保留（系统提示 + 最早几轮）
[中间一大段]  ← 压成一条摘要（summary）
[tail 尾部]   最近几条原样保留（最新提问 + 最新回答）
```

拼回去就是 `head + 一条 summary 消息 + tail`。但 summary 自己也得有个角色（`user` 或 `assistant`），而整个列表又绝不能连续两个相同角色——这就引出矛盾。

**核心矛盾：summary 该戴哪个"帽子"？** 举个具体例子：

```
[system]    系统提示            ← head 末条
[ ??? ]     摘要                ← user 还是 assistant？
[user]      问3（最新提问）      ← tail 首条
[assistant] 答3
```

- 给 `user` 帽子 → `system, user, **user**, assistant` → ❌ 两个 user 连着
- 给 `assistant` 帽子 → `system, assistant, user, assistant` → ✅ 完美交替

代码按左右邻居动态挑帽子，挑那个两边都不撞的（`context_compressor.py:2862`）。大多数情况这步就解决。

**万一两个帽子都撞，就把摘要"塞进"邻居肚子里。** 看这个例子：

```
[user]      问X            ← head 末条
[ ??? ]     摘要            ← 帽子？
[assistant] 答3            ← tail 首条
```

- `user` → `user, **user**, assistant` → ❌（和上面问X撞）
- `assistant` → `user, **assistant**, assistant` → ❌（和下面答3撞）

两头堵死。**解法：让摘要别独立站——把摘要文本并进下面那条消息（答3）的正文里**，用明显的分隔符包好：

```
[user]      问X
[assistant] ──[历史摘要，仅供参考，不是新指令]──
             之前讨论过 A、完成了 B、卡在 C……
             ──[摘要结束，请回答下面这条]──
             答3 原来的内容……
```

这样角色序列 `user → assistant` **完全没变**，自然不违反交替规则。

> **一句话总结**：压缩 = head + 摘要 + tail 三段拼接；摘要必须戴角色帽但又不能撞邻居——代码先挑不撞的帽子，挑不出来就把摘要塞进邻居正文，让角色序列原封不动。无论哪种，`user / assistant` 严格交替都不破。

#### （4）压缩用哪个模型

**默认用专门的"辅助模型"，不是主对话模型**（`agent/auxiliary_client.py`）。这样压缩不占主模型额度、可挑便宜快速的模型。配在 `config.yaml` 的 `auxiliary.compression`。辅助模型不可用或失败时，**回退主模型重试一次**。启动时还有个检查：辅助模型窗口太小（< 6.4 万）直接拒绝启动压缩。

#### （5）压缩和"prompt cache 神圣不可变"怎么共处

CLAUDE.md 说会话中途不能改上下文（会击穿 cache、每轮成本翻倍），**唯一例外就是压缩**。代码处处体现"压缩是最后手段"：

- 触发层就有冷却期 + 反抖动在**主动抑制**触发。
- 优先用 provider 实报的真实 token 数（粗估含工具 schema 会高估），会话没真正长大就压着不压。
- 压缩刚跑完那轮，下一轮预检**强制跳过**，等真实 token 重新校准。
- 压缩后**重置而非续接**：系统提示被重建（因为往里加了"[部分历史已压缩]"提示语）、token 状态机全部清零——cache 失效就失效，下轮重新预热。

#### （6）压缩完的消息存哪 / resume 看到哪个版本

**会落库，且 resume 看到的是压缩后版本。** 默认 `compression.in_place: true`（`hermes_cli/config.py`）：旧轮次被**软归档**（`active=0`，还在盘上、可被 `session_search` 全文搜到），压缩后的新活跃集 `active=1` 用**同一个 session_id** 插入。加载活跃上下文时过滤 `active=1`，所以 resume 只看到压缩后版本；原始轮次仍可被搜索到、可恢复。

#### （7）压缩失败怎么办

按错误类型分（模型没找到 / 超时 / JSON 错误 → 切主模型重试；401/403 鉴权 → 不重试）。摘要失败后，只有三种情况**冻结会话不动**（配置 `abort_on_summary_failure`、鉴权坏、网络断）；**否则走确定性兜底**：本地不调 LLM，直接拼一个结构化 handoff（提取历轮提问、做过的事、文件路径、错误文本，全脱敏）——绝不让用户卡在高上下文里干瞪眼。单次错误恢复最多压 3 次；会话内累计压到第 2 次会警告"精度可能下降，建议 `/new`"。

#### （8）怎么开关 / 调阈值

`config.yaml`：

```yaml
compression:
  enabled: true            # 总开关；false 时超限直接报错，不压
  threshold: 0.50          # 触发比例（有效输入预算的 %）
  target_ratio: 0.20       # 尾部保留预算 = threshold × 此值
  protect_last_n: 20       # 尾部至少保留多少条（token 预算优先）
  abort_on_summary_failure: false
  in_place: true           # 不轮转 session_id，原地软归档

auxiliary:
  compression:             # 压缩用的辅助模型
    provider: auto
    model: ""
    timeout: 120
```

**没有 `HERMES_COMPRESSION_*` 环境变量**（符合 ".env 只放 secret" 的规矩）。改用 `hermes config set compression.threshold 0.7`。

> **一句话总结**：压缩是受严格管控的"最后手段"——token 到阈值才触发，先免费剪枝再调辅助模型摘要，拼回去时靠"挑帽子 / 塞邻居"保住角色交替，失败有确定性兜底，落库后 resume 只看压缩版。

## 5. 关键设计决策（摘要）

完整 9 个决策见 [`改造计划.md`](改造计划.md)。最影响架构的：

1. **单租户每客户独立部署** + 共享实例多用户 → 代码无租户概念，靠 user_id 隔离。
2. **沙盒默认**（docker-only）→ agent 够不到宿主；`features.host_terminal` 是唯一 opt-in。
3. **DB 只读在 GRANT 层** → 工具不限制 SQL，数据库用只读用户拒绝写。
4. **能力面分层**：db_query 核心工具；算法/数据接口按规则做核心工具或 MCP server；专门 agent（如 nl2sql）做 MCP 包裹服务；临时并行用 delegate_task。
5. **持久记忆本地化**（无云 provider）+ 按 user_id 隔离。
6. **prompt cache 神圣**（继承自 Hermes）→ 不在对话中途改上下文/换工具集/重建系统提示。
7. **硬分叉不跟踪上游** → 安全修复手动 cherry-pick（见 UPSTREAM.md）。
