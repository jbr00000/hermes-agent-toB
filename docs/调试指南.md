# 如何调试 agent（main_debug.py）

> 配合 [`项目架构详解.md §4 核心运行机制`](项目架构详解.md) 阅读——你在断点处看到的就是 §4 讲的那套多轮循环 / 上下文压缩机制。

## 1. 这个脚本是干嘛的

一句话：**在 VSCode 里把 agent 当 main 函数跑一遍，断点单步进框架代码。**

它跑一个固定真实任务（读一个 Excel → 总结 → 写 txt 存本地），覆盖完整 agent 路径：解析 provider → 构造 `AIAgent` → 对话循环 → 工具调用 → 落盘。它**不走 `server/`、不走 `hermes` CLI**，直接 `AIAgent(...)` + `agent.run_conversation(prompt)`，所以你看到的是**纯净的 agent 核心**，不被 HTTP/SSE/鉴权层干扰。

## 2. 什么时候用它

- 想搞清楚「agent 内部到底怎么转的」——单步进对话循环。
- 工具调用出 bug——看 `tool_executor` 怎么 dispatch、结果怎么变回消息。
- 上下文压缩行为不对——看 `context_compressor`。
- 新 provider / 模型接入——看 `runtime_provider` 解析对不对。
- 想复现某一轮的问题，但不想起整个 server。

## 3. 跑之前的准备

1. **配好凭据**：在 `.hermes-dev/.env` 里放 provider 的 key（如 `DEEPSEEK_API_KEY=...`）。脚本固定用 `HERMES_HOME=.hermes-dev`，不污染正式环境。
2. **配好 `config.yaml`**：`agent` / `providers` 段（脚本读的就是这份，和正常 `hermes` 运行同一份）。
3. **依赖装齐**：`uv pip install -e ".[all,dev]"`。

## 4. 在 VSCode 里怎么用（推荐）

1. 装好 Python 扩展。
2. 打开 `main_debug.py`，选 Hermes 的 Python 解释器（带 hermes 依赖的那个 venv）。
3. 改顶部 `DEBUG_*` 常量（任务文件、模型、toolset——见下表）。
4. 在想看的函数里打断点（推荐起点见 §6）。
5. 按 **F5** 或 "Run and Debug" → 选 "Python: Current File"。
6. 命中后：**F11 (Step Into)** 一层层进；Locals / Watch 看变量；Debug Console 里试表达式（比如 `len(messages)`、`[m["role"] for m in messages]`）。

## 5. 命令行跑（不看断点，只看 trace）

```bash
python main_debug.py
```

跑完去 `test_outputs/debug_runs/<时间戳>/` 看产物（见 §7）。

## 6. 推荐断点起点（对照 §4 机制）

| 想看什么 | 打断点 / Step Into 到哪 |
|---|---|
| **对话主循环** | `main_debug.py` 的 `run_debug_conversation` → Step Into → `agent/conversation_loop.py` 的 `run_conversation`（循环体，对照 §4.1） |
| **消息列表怎么长出来** | 同上循环体里看 `messages` 列表（每轮 append，对照 §4.1(2)） |
| **工具怎么并发执行** | `agent/tool_executor.py` 的 `execute_tool_calls_concurrent` |
| **工具结果怎么变回消息** | `agent/tool_dispatch_helpers.py` 的 `make_tool_result_message`（role=tool，对照 §4.1(3)） |
| **上下文压缩何时触发 / 怎么拼回** | `agent/context_compressor.py` 的 `compress()`；摘要边界选帽子逻辑在同文件 ~2862 行（对照 §4.2(3)） |
| **provider / 凭据怎么解析** | `hermes_cli/runtime_provider.py` 的 `resolve_runtime_provider` |
| **消息怎么发 API / 流式收 token** | `agent/chat_completion_helpers.py` 的 `interruptible_streaming_api_call` |

## 7. 改哪些常量（`main_debug.py` 顶部）

| 常量 | 意思 | 改它来干嘛 |
|---|---|---|
| `DEBUG_XLSX_PATH` | 任务输入文件 | 换成你想测的文档 |
| `DEBUG_OUTPUT_PATH` | 期望产物路径 | 换个输出位置 |
| `DEBUG_PROVIDER` / `DEBUG_MODEL` | 用哪个 provider / 模型 | 测不同模型 |
| `DEBUG_TOOLSETS` | `None`=全部内置工具；或列如 `["terminal","file"]` | 缩小工具面、定位某工具 |
| `DEBUG_MAX_ITERATIONS` | 单 agent 最多调模型次数（默认 90） | 调大看长任务，调小快速失败 |
| `DEBUG_SKIP_MEMORY` / `DEBUG_SKIP_CONTEXT_FILES` | 跳过记忆 / 上下文文件注入 | 排除干扰、最小复现 |
| `DEBUG_HERMES_HOME` | 用哪个 HERMES_HOME（默认 `.hermes-dev`） | 切换隔离环境 |

## 8. 产物怎么看

每次跑会在 `test_outputs/debug_runs/<时间戳>/` 下生成：

- `debug_prompt.txt` —— 发给 agent 的任务提示词。
- `debug_final_response.txt` —— agent 的最终回答。
- `debug_result.json` —— 完整 `result`（messages / api_calls / completed）+ `verification`（产物文件大小/行数/预览）+ `trace`（每个工具开始/完成、状态、步骤的时间线）。**写盘前已脱敏**（api_key/authorization 等 → `[REDACTED]`），大字符串也会截断。

排查「为什么没生成 txt」：先看 `verification.exists`，再看 `trace` 里 agent 调了哪些工具、有没有报错。

## 9. 脚本自动设的环境变量（一般不用管）

- `HERMES_HOME=.hermes-dev` —— 用开发 home，不动正式。
- `HERMES_FEATURE_HOST_TERMINAL=1` —— 开主机终端权限（任务要用 shell 读 Excel）。
- `HERMES_YOLO_MODE=1` / `HERMES_ACCEPT_HOOKS=1` —— 跳过交互确认。
- `stdout`/`stderr` 重设 UTF-8 —— Windows 下防中文乱码。

## 10. 一个最小调试场景：看 agent 怎么决定调工具

1. 断点打在 `run_debug_conversation` 里的 `result = agent.run_conversation(prompt)`。
2. F5 跑，命中。
3. F11 Step Into → 进 `conversation_loop.run_conversation`。
4. 在 `while ...:` 循环体停下，看 `messages` 列表怎么一轮轮长出来（对照 §4.1(2)）。
5. 模型返回 `tool_calls` 时，Step Into `_execute_tool_calls`，看它怎么并发跑、结果怎么按序填回。
6. 跑完打开 `debug_result.json`，`trace` 数组里有每个工具的开始/完成时间戳。

> **一句话总结**：`main_debug.py` 是「把 agent 当 main 函数跑」的调试入口——改顶部 `DEBUG_*` 常量、按 §6 表选断点位置、F5 开跑；断点处看到的东西，对应 `项目架构详解.md §4` 讲的机制。
